# ADR-120: Rust CI Uses Contract Fixtures and Live-Corpus Invariants

## Status

Accepted

## Context

The native engine's CI accumulated large expected-output files generated from
the live `rac/` corpus by the Python implementation. Any documentation change
could invalidate megabytes of snapshots, and the failure instructed
maintainers to regenerate them from an implementation we intend to retire.

This confused two different jobs. Compatibility needs small, stable examples
whose expected behavior is independent of repository churn. The live RAC
corpus needs continuous evidence that it is healthy, but not frozen bytes.

## Decision

`rac-spec` is the sole authority for cross-engine compatibility fixtures.
Those fixtures are small, language-neutral, deterministic, and focused on
public contracts.

Normal Rust CI contains no live-corpus expected-output snapshots and does not
invoke Python as an oracle. It proves:

- the Rust unit and bounded internal regression suites pass;
- the vendored artifact registry matches `rac-spec`;
- the Rust executable reproduces every `rac-spec` conformance case;
- the live RAC corpus is valid and deterministic;
- cache-on and cache-off outputs are equal; and
- a completed corpus mutation is visible through the cache freshness path.

The full-corpus classification, Markdown, validation, scoring, and retrieval
snapshots are removed from rac-core together with their generators. Bounded
synthetic parser vectors may remain as internal Rust regression data; they are
not compatibility authority.

Python comparison survives only as one bounded retirement-certification job:
the frozen Python and Rust executables must both reproduce the same `rac-spec`
suite. It does not fetch historical feature branches, compare the live corpus,
or run the former CLI/MCP parity matrices.

The retirement-certification job is deleted once the Rust authority cutover
gates are accepted: Rust passes the complete `rac-spec` tier, the live-corpus
invariants, and the native performance/correctness stack selected for cutover.
After that point Python is absent from normal Rust CI and from retirement CI.

## Consequences

RAC documentation can evolve without rewriting implementation-derived golden
files. Compatibility changes must be proposed in `rac-spec`, making the
contract reviewable before either engine changes.

Live-corpus checks become stronger in the dimensions that matter operationally:
freshness and cache neutrality are exercised rather than inferred from static
snapshots. They deliberately do not assert that the corpus always returns the
same search ranking as its contents evolve.

The bounded Python job gives the cutover a clear, removable seam. Python is no
longer a transitive authority for every Rust test.

## Alternatives Considered

### Regenerate every live-corpus snapshot after each documentation change

Rejected. It produces large review noise and preserves Python as the de facto
authority indefinitely.

### Bless snapshots generated by Rust

Rejected. Self-generated expected outputs do not establish compatibility and
would still freeze an intentionally evolving corpus.

### Remove Python comparison immediately

Rejected for this step. One bounded `rac-spec` certification remains until the
recorded Rust cutover gates pass, then is removed explicitly.
